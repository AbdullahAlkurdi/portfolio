## Context

Rizen was born from a fundamental observation: existing habit trackers treat personal growth as a checklist. Users check boxes, watch streaks grow, but receive no meaningful insight into their patterns, no adaptive coaching, and no holistic view of their life.

The engineering challenge was to build a system that could understand context, adapt to individual lifestyles, and evolve with user goals — while maintaining offline-first reliability and a clean, maintainable architecture.

## Problem

Existing solutions share three critical failures:

**Superficial tracking.** Most apps track binary completion — did you or didn't you? They ignore nuance, context, and the qualitative aspects of habit formation. A meditation session isn't just "done" or "not done"; its quality, duration, and consistency all matter.

**No adaptive intelligence.** Users plateau because the app treats everyone identically. No personalization, no pattern recognition, no proactive suggestions based on user behavior.

**Fragmented experience.** Health, productivity, learning, finance, relationships — users manage these domains across multiple apps. Rizen aims to unify them under a single coherent system.

## Goals

1. **Unified life OS** — One app for habits, routines, goals, and analytics across all life domains
2. **AI-powered coaching** — Gemini-driven insights that adapt to individual patterns
3. **Offline-first** — Full functionality without connectivity
4. **Clean Architecture** — Maintainable, testable code with clear layer separation
5. **Cross-platform** — Consistent experience across iOS and Android

## Architecture

The system follows Clean Architecture with three distinct layers, each with strict dependency rules. The domain layer contains zero framework dependencies — pure Dart business logic that can be tested without Flutter.

<Callout variant="success">
  **Key decision**: Isolate AI behind a repository interface from day one. This allows switching from an initial rule-based engine to Gemini without touching domain logic.
</Callout>

### Presentation Layer

Flutter widgets with BLoC pattern for state management. Each feature (habits, routines, analytics, coaching) has its own module with screens, widgets, BLoCs, and routes. GoRouter handles navigation.

### Domain Layer

Pure Dart classes defining entities, repository interfaces, and use cases. Every business rule is expressed here with no Flutter dependency.

### Data Layer

Repository implementations that coordinate between domain models and data sources. DTOs handle serialization. Data sources abstract Firebase, local storage, and platform APIs.

### Infrastructure

Firebase for cloud sync and auth, Isar for local persistence, Gemini for AI, platform channels for native features.

## Key Decisions

### Why BLoC?

Predictable state machines. Every state transition is explicit. Testing is straightforward — feed events, assert states.

### Why Isar over Hive?

Isar provides type-safe queries, relations, and indexes with minimal overhead. The offline-first architecture required a local database that could handle complex queries without performance degradation.

### Why Gemini?

Gemini's multimodal capabilities allow processing not just text but also user-uploaded images (journal scans, progress photos) within a single model.

## Current Status

Rizen is under active development. Core architecture is implemented with Clean Architecture layers, BLoC state management, and Firebase integration. AI coaching with Gemini is in progress.

## Lessons Learned

1. **Invest in repository interfaces early.** The AI repository abstraction saved months when switching from rule-based to ML-based coaching approaches.
2. **Domain purity pays dividends.** The ability to run domain tests without Flutter makes the feedback loop faster.
3. **Don't over-abstract early.** Some layers went through unnecessary indirection before the architecture stabilized.
